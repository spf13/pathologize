## 2024-04-14 - Inline Regex and Repeated Allocation Anti-pattern
**Learning:** Found two significant performance bottlenecks in `pathological.go`: compiling `regexp.MustCompile` inline inside `removeTrailing`, and repeatedly calling `strings.Split` on a constant string inside `removeReservedNames`. Because these functions execute on every single directory segment of a parsed path, the repeated heap allocations and regex engine initialization caused massive cumulative overhead (e.g. over 3000ns per operation for the regex alone). Replacing the inline regex with the native `strings.TrimRight` and pre-calculating the `strings.Split` array globally eliminated all heap allocations and improved execution speed of these functions by ~5x and ~150x, respectively.
**Action:** When auditing string-heavy string sanitization libraries like `pathologize`, proactively search for inline `regexp.MustCompile` calls and `strings.Split` operations on static constants. Always hoist these to package-level global variables or replace them entirely with native standard library `strings` functions for massive free performance wins.
