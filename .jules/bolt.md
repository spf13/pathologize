## 2024-03-29 - Inline Regex Compilation vs Native Strings Functions
**Learning:** In Go, repeatedly compiling a regular expression inline (e.g., `regexp.MustCompile` inside a function called frequently like `removeTrailing`) introduces a massive performance bottleneck due to continuous heap allocations and parsing overhead. Even outside of inline compilation, replacing simple regexes with native `strings` equivalents yields significant speedups. Notably, Go's regex `\s` matches exactly `[\t\n\f\r ]` (it explicitly excludes the vertical tab `\v`), so an exact mapping with `strings.TrimRight` is possible and much faster.
**Action:** Always prefer native `strings` functions (like `strings.TrimRight`, `strings.EqualFold`, etc.) over `regexp` for simple character manipulations. If regex is absolutely necessary, pre-compile it as a package-level variable instead of inside a function.
