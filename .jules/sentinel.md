## 2024-06-08 - Path Sanitization Order of Operations Vulnerability
**Vulnerability:** The path sanitization library truncates filenames to 255 characters AFTER performing semantic validation like removing trailing dots and spaces (`removeTrailing`). An attacker can craft a 256-character filename ending with a dot/space followed by a benign character (e.g., `CON. ` + `a`*250 + `.x`). The semantic validation ignores the hidden dot/space because it isn't at the end of the string yet. Then, truncation strips the final benign character (`x`), exposing the trailing dot/space in the final output. On Windows, trailing spaces and dots are ignored, allowing this bypass to create reserved device names (e.g. `CON`) or other unsafe files.
**Learning:** Always perform length bounding/truncation BEFORE evaluating semantic meaning, checking for reserved names, or trimming trailing spaces/dots. Truncating after these validation steps can create new string suffixes that bypass the filter logic.
**Prevention:** Reorder the sanitization pipeline so that `truncateFilename` happens immediately after removing invalid characters, but strictly before any semantic checks like `removeTrailing` or `removeReservedNames`.
